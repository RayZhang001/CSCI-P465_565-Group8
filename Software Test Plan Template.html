1. Overview

This section provides an overview of the testing approach used to verify the software product.

1.1 Test Objectives

Provide a description of the objectives of the test.  That is, what is it that is to be accomplished by the testing effort.

The testing effort wants to accomplish complete coverage of all features of our project.  That is, we want to make sure the results users are experiencing are in-line with the results we expect for every component of the project.

1.2 Test Environment

Describe the environment that will be used to test the software product.  Include a description of the hardware and operating system of all machines involved in the test.  Describe the networking between those machines (if your product requires network support).  Describe any support software packages that need to be installed/available/running on the machine for the software to operate.  Comment on whether the test environment is the same as the operational environment, and descibe any differences.

Our testing environment will consist of multiple computers, with some running Mac OS and the others running Windows 10. These machines are powerful enough for the usages of testing, so there will not be an issue with hardware. Each machine will be running the tests independently, so there is no need for network communication between them. For the tests to be ran, there will be a couple software packages that need to be installed, these include:
  • Node.js
  • MongoDB
  • Vue.js
  • Express.js
The testing environment will be the same as the operational environment, as we want to assure the features we are testing will work in the production environment.

1.3 Test Personnel

Describe the personnel involved in the testing effort, and the roles that they will fill.  Describe the level of user involvement in the testing process (if any).

Every member of the team will have an integral part of the testing effort. There will be no set in stone roles, as having the team experience all areas of testing will allow for better coverage and testing techniques. There will be no user involvement in the testing process, due to the fact we believe we can test, and fix any potential issues that may arise.

1.4 Acceptance Criteria

Under what conditions will the software be considered acceptable to deliver.  For example, how many errors will you tolerate before the testing is abandoned to correct the errors and try again?  Or, how many of the requirements need to be verified before the software is considered deliverable.  Identify in this section an acceptance criteria that is feasible for your development team and acceptable for your customer/user.

We will not accept any errors during the testing phase. If we run a test and confirm that it is indeed our implementation that is wrong, we will immediately fix it to pass the tests. Every single one of the requirements must be met for our product to be considered deliverable. Anything less than perfection will not be permitted in the development of our product. The acceptance criteria that will be feasible for both our customers and us is that of a product which fills all of the requirements the customer listed.

1.5 Noted Omissions

Identify any software specification statements that are not intended to be verified by this test, and justify the exclusion of these specifications.

All software specification statements will be covered by this test, as we need to be 100% sure that our product is up to par.
